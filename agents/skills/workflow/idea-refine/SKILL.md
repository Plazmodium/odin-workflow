# idea-refine

## When To Use

- The user has a rough concept, not a concrete request.
- Goals, users, constraints, or success criteria are unclear.
- Several possible solutions could satisfy the request.

## Workflow

1. Restate the idea in one sentence.
2. Identify the target user, problem, and desired outcome.
3. List constraints, non-goals, risks, and unknowns.
4. Generate 2-3 plausible solution shapes.
5. Compare tradeoffs briefly.
6. Recommend the smallest viable direction.
7. Ask only the questions needed to unblock a spec or implementation.

## Anti-Rationalization

| Excuse | Rebuttal |
| --- | --- |
| "The idea is obvious." | Obvious ideas often hide conflicting success criteria. |
| "I should start coding to discover the shape." | Discovery code becomes accidental product code. Clarify first. |
| "Asking questions slows things down." | One precise question is cheaper than building the wrong thing. |

## Verification

- The proposal names the user, problem, outcome, constraints, and non-goals.
- Open questions are explicit.
- The recommended direction is smaller than the broadest possible solution.

## Exit Criteria

- The user or agent can write a spec with clear acceptance criteria.
